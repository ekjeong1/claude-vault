## 버전 관리 프로세스

### 1. 업데이트 주기

| 변경 유형 | 기록 시점 | 담당 | 승인 |
|-----------|----------|------|------|
| **개념 추가** | 새 개념 파일 생성 시 | 작성자 | 즉시 |
| **개념 확장** | 주요 내용 추가 (2배+ 증가) | 작성자 | 즉시 |
| **개념 수정** | 기존 내용 변경 | 작성자 | 즉시 |
| **구조 변경** | 폴더, 인덱스 재구성 | 관리자 | 검토 필요 |
| **메타 문서** | 가이드, 템플릿 변경 | 관리자 | 검토 필요 |

### 2. 기록 방법

**각 변경사항은 다음 정보를 포함**:

```markdown
### YYYY-MM-DD: [변경 제목]

**변경 유형**: [추가/확장/수정/삭제/재구성]
**영향 범위**: [Part 번호 또는 "전체"]
**불변량 관련**: [어떤 불변량 준수를 위한 변경인가]

**변경 내용**:
- 구체적 변경 사항 1
- 구체적 변경 사항 2

**변경 사유**:
- 왜 이 변경이 필요했는가

**검증 결과**:
- [ ] 불변량 체크리스트 통과
- [ ] 크로스링크 업데이트
- [ ] 인덱스 파일 업데이트

**영향**:
- 사용자에게 미치는 영향
- 다른 개념과의 연관성
```

### 3. 불변량 준수 확인

모든 변경사항은 5가지 불변량을 준수해야 함:

| 불변량 | 확인 질문 | 통과 기준 |
|--------|-----------|-----------|
| **1. 복잡성 단순화** | 이 변경이 시스템을 더 단순하게 만드는가? | 복잡도 유지 또는 감소 |
| **2. 데이터 기반** | 측정/재현 가능한가? | 정량적 지표 포함 |
| **3. 구조적 사고** | 근본 원인을 다루는가? | 일회성이 아닌 구조 개선 |
| **4. 점진적 개선** | 지속적 학습이 가능한가? | 업데이트 프로세스 명시 |
| **5. 실용주의** | 즉시 적용 가능한가? | 실전 도구 포함 |

### 4. 롤백 정책

**언제 롤백하는가**:
- 불변량 2개 이상 위반 시
- 사용자 피드백 부정적 (만족도 <3.0/5)
- 시스템 복잡도 20% 이상 증가 시

**롤백 방법**:
```bash
# Git 히스토리에서 이전 버전 복원
git log --oneline  # 커밋 찾기
git checkout [commit-hash] -- [file-path]
git commit -m "Rollback: [사유]"
```
