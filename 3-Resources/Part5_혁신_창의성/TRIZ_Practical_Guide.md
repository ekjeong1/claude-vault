## 개요

TRIZ(발명적 문제 해결 이론)를 비즈니스 문제에 실전 적용하는 가이드. 40가지 발명 원리를 3-5개 핵심 원리로 압축하고, 단계별 워크시트로 즉시 사용 가능하게 만든다.

**불변량 준수**:
- ✅ **복잡성 단순화**: 40개 원리 → 5개 핵심 원리로 시작
- ✅ **데이터 기반**: 케이스 스터디로 효과 검증
- ✅ **구조적 사고**: 모순의 구조적 해결
- ✅ **점진적 개선**: 작은 문제부터 적용
- ✅ **실용주의**: 워크시트로 오늘 바로 사용

## TRIZ 핵심 원리

### TRIZ란?

수백만 개의 특허를 분석해 발견한 **혁신의 반복 패턴**. 혁신은 무작위가 아니라 구조를 따른다.

### 핵심 통찰

**"모든 혁신은 모순(Contradiction)의 해결이다"**

- **기술적 모순**: A를 개선하면 B가 악화
- **물리적 모순**: 동일한 것이 상반된 속성을 동시에 가져야 함

---

## 5가지 핵심 발명 원리 (비즈니스용)

TRIZ의 40가지 원리 중 비즈니스에서 가장 자주 쓰이는 5가지:

### 1. 분리 (Separation)

**원리**: 상충하는 요구사항을 시간/공간/조건으로 분리

**비즈니스 적용**:
- **시간 분리**: 피크 타임과 오프 타임에 다른 가격/서비스
- **공간 분리**: 온라인과 오프라인 채널 차별화
- **조건 분리**: 고객 세그먼트별 다른 제품

**예시**:
```
모순: 높은 품질 vs 낮은 가격
해결: 프리미엄 라인 + 이코노미 라인 (공간 분리)
결과: 두 시장 동시 공략
```

---

### 2. 사전 조치 (Prior Action)

**원리**: 문제가 발생하기 전에 미리 해결

**비즈니스 적용**:
- **사전 품질 검증**: 출시 전 베타 테스트
- **사전 고객 교육**: 온보딩 프로그램
- **사전 리스크 관리**: 시나리오 플래닝

**예시**:
```
모순: 빠른 출시 vs 품질 보장
해결: 출시 전 3주간 집중 베타 테스트 (사전 조치)
결과: 출시 후 버그 90% 감소
```

---

### 3. 중개자 (Intermediary)

**원리**: 직접 충돌 대신 중간 매개체 사용

**비즈니스 적용**:
- **플랫폼 모델**: 직접 거래 대신 플랫폼 중개
- **프로세스 버퍼**: 팀 간 대기열로 충돌 완화
- **데이터 기반 의사결정**: 주관 대신 데이터 중개

**예시**:
```
모순: 빠른 요청 처리 vs 품질 유지
해결: 우선순위 큐 시스템 (중개자)
결과: 긴급 요청 즉시 처리, 일반 요청은 품질 검증 후 처리
```

---

### 4. 비대칭 (Asymmetry)

**원리**: 대칭을 깨고 비대칭 구조로 전환

**비즈니스 적용**:
- **차별화된 역할**: 모든 팀원이 동일 → 전문화된 역할
- **비대칭 가격**: 단일 가격 → 가치 기반 가격
- **선택적 집중**: 모든 기능 개선 → 핵심 3개만 집중

**예시**:
```
모순: 모든 고객 만족 vs 제한된 자원
해결: 상위 20% 고객에 80% 자원 집중 (비대칭)
결과: 매출의 80% 창출하는 고객 만족도 95% 달성
```

---

### 5. 피드백 (Feedback)

**원리**: 결과를 입력으로 다시 순환

**비즈니스 적용**:
- **데이터 루프**: 실험 → 측정 → 학습 → 개선
- **고객 피드백**: 사용 데이터로 제품 개선
- **베이즈 업데이트**: 새 정보로 전략 조정

**예시**:
```
모순: 완벽한 제품 vs 빠른 출시
해결: MVP 출시 → 피드백 수집 → 점진적 개선 (피드백)
결과: 3개월 만에 시장 적합 제품 달성
```

## 4단계 TRIZ 워크시트

### Step 1: 문제 정의

**질문**:
1. 우리가 해결하려는 문제는 무엇인가?
2. 왜 이 문제가 어려운가?
3. 무엇과 무엇이 충돌하는가?

**예시**:
```
문제: 고객 응답 시간을 줄이고 싶다
어려움: 빠른 응답을 하려면 품질이 떨어진다
충돌: 속도 ↔ 품질
```

---

### Step 2: 모순 식별

**기술적 모순**:
- A를 개선하면 → B가 악화
- 예: "응답 속도를 높이면 → 답변 품질 저하"

**물리적 모순**:
- 동일한 것이 상반된 속성 필요
- 예: "고객 지원팀은 많아야 하고(빠른 응답) 적어야 한다(비용 절감)"

**워크시트**:
```
[ ] 기술적 모순: ______를 개선하면 ______가 악화
[ ] 물리적 모순: ______는 동시에 ______과 ______여야 함
```

---

### Step 3: 원리 선택

위 5가지 원리 중 적용 가능한 것은?

**체크리스트**:
- [ ] **분리**: 시간/공간/조건으로 나눌 수 있나?
- [ ] **사전 조치**: 문제 발생 전에 미리 해결 가능한가?
- [ ] **중개자**: 중간 매개체로 완충할 수 있나?
- [ ] **비대칭**: 균등 배분 대신 선택적 집중 가능한가?
- [ ] **피드백**: 결과를 측정하고 반복 개선할 수 있나?

**팁**: 2-3개 원리를 조합하면 더 강력함

---

### Step 4: 솔루션 생성 및 검증

**솔루션 후보**:
1. 선택한 원리를 우리 문제에 적용하면?
2. 구체적으로 무엇을 어떻게 바꿀 것인가?
3. 예상 결과는?

**검증**:
- [ ] 모순이 해결되는가?
- [ ] 새로운 문제가 생기지 않는가?
- [ ] 측정 가능한가?
- [ ] 실행 가능한가?

## 케이스 스터디 1: 조직 병목 해결 (Part 1 연결)

### 문제

**상황**: [[Part1_조직_인력/그래프 이론과 중심성|그래프 이론]] 분석 결과, 디자인 팀장 1명이 모든 승인 담당 (betweenness centrality 최고)

**모순**:
- 팀장이 승인해야 품질 보장 (집중)
- 팀장이 병목이라 프로젝트 지연 (분산)

### TRIZ 적용

**원리 1: 분리 (조건)**
- 중요도에 따라 승인 경로 분리
- 고위험: 팀장 승인 필수
- 저위험: 시니어 디자이너 승인 가능

**원리 2: 사전 조치**
- 명확한 디자인 가이드라인 작성
- 80%는 가이드라인만으로 해결
- 20%만 실제 승인 필요

**원리 4: 비대칭**
- 모든 승인을 동등하게 보지 않음
- 전략적 프로젝트 20%에 팀장 집중
- 일상 업무 80%는 위임

### 결과

**측정 가능한 성과**:
- 승인 대기 시간: 5일 → 1일 (80% 감소)
- 프로젝트 지연: 월 15건 → 월 3건 (80% 감소)
- 팀장 업무 부담: 주 40시간 → 주 25시간 (37% 감소)
- 품질 유지: 결함률 변화 없음

**불변량 체크**:
- ✅ 구조적 사고: 병목 구조 자체를 변경
- ✅ 데이터 기반: 승인 대기 시간 측정
- ✅ 실용주의: 측정 가능한 비즈니스 성과

## 케이스 스터디 2: 프로세스 재설계 (Part 2 연결)

### 문제

**상황**: [[Part2_프로세스_운영/시스템 사고와 피드백 루프|시스템 사고]] 분석 결과, 압박 → 품질 저하 → 재작업 → 더 큰 압박 (강화 루프)

**모순**:
- 빠르게 해야 함 (속도)
- 제대로 해야 함 (품질)

### TRIZ 적용

**원리 1: 분리 (시간)**
- 빠른 작업과 품질 작업을 시간으로 분리
- 1차 작업: 빠른 프로토타입 (2일)
- 2차 작업: 품질 검증 및 개선 (3일)

**원리 2: 사전 조치**
- 작업 시작 전 체크리스트 작성
- 필수 요구사항 명확화
- 예상 함정 사전 식별

**원리 5: 피드백**
- 매 단계 품질 검증
- 문제 발견 시 즉시 수정 (재작업 비용 최소화)
- 축적된 데이터로 체크리스트 개선

### 결과

**측정 가능한 성과**:
- 재작업률: 35% → 8% (77% 감소)
- 평균 작업 시간: 8일 → 5일 (37% 단축)
- 품질 점수: 72점 → 89점 (24% 향상)
- 팀 스트레스: 설문 3.2/5 → 4.1/5 (28% 개선)

**불변량 체크**:
- ✅ 구조적 사고: 악순환 피드백 루프를 균형 루프로 전환
- ✅ 점진적 개선: 체크리스트가 시간이 지나며 개선됨
- ✅ 실용주의: 실제 작업 시간과 품질 모두 향상

## 케이스 스터디 3: 제품 개발 (독립 사례)

### 문제

**상황**: 신제품 개발 시 기능이 많아질수록 출시 지연

**모순**:
- 많은 기능 = 더 나은 제품 (확장)
- 빠른 출시 = 시장 선점 (압축)

### TRIZ 적용

**원리 4: 비대칭**
- 모든 기능을 동등하게 보지 않음
- 핵심 3개 기능만 MVP에 포함
- 나머지는 우선순위 백로그

**원리 5: 피드백**
- MVP 출시 → 사용 데이터 수집
- 실제로 사용되는 기능만 개발
- 3주 스프린트로 점진적 추가

**원리 2: 사전 조치**
- 기능 개발 전 고객 인터뷰
- Fake Door 테스트로 수요 검증
- 개발 전 수요 확인

### 결과

**측정 가능한 성과**:
- 출시 시간: 9개월 → 3개월 (67% 단축)
- 사용되는 기능: 40% → 85% (활용률 112% 증가)
- 개발 비용: $500K → $200K (60% 절감)
- 고객 만족도: 3.5/5 → 4.3/5 (23% 향상)

**불변량 체크**:
- ✅ 복잡성 단순화: 15개 기능 → 3개 핵심 기능
- ✅ 데이터 기반: 사용 데이터로 개발 우선순위 결정
- ✅ 실용주의: 실제 출시와 비용 절감
